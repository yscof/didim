# Codex Project Context: 디딤

이 저장소의 목적은 금융 챌린지 앱 `디딤`을 제품, 기획, 마케팅, AI 설계, 개발까지 함께 고도화하는 것이다. Codex는 코드만 작성하지 말고, 아래 제품 원칙과 문서 구조를 항상 참고해 프로젝트 일관성을 유지한다.

## 제품 정체성

- 이름: 디딤
- 한 줄 설명: Challenge-Based Learning으로 사회초년생의 첫 자산을 굴리는 AI 금융 행동 코치
- 대상: 첫 월급 이후 자산관리 기준이 부족한 사회초년생 1~3년차
- 문제: 금융 정보는 많지만 개인 상황에 맞는 우선순위와 실행까지 이어지는 구조가 부족하다.
- 목적: 자산 격차를 줄이고 금융 리터러시를 실행 경험으로 높인다.

## 핵심 차별점

- 공부 앱이 아니라 실제 금융 행동을 완수하게 하는 앱이다.
- 매주 하나의 실천 가능한 챌린지를 제안한다.
- 챌린지는 Engage, Investigate, Act, Reflect 흐름을 따른다.
- 금융 단계 진단은 룰 기반 우선순위 모델을 사용하고, 설명과 코칭은 LLM이 담당한다.
- 정확성이 중요한 정보는 공식 자료 기반 RAG와 출처 표기를 기본값으로 한다.

## 5단계 자산 형성 모델

1. 비상금
2. 청약·연금
3. 적금·예금
4. 투자 기초
5. 자산 분산

사용자를 한 칸에 고정하는 분류가 아니라, 가장 먼저 비어 있는 토대를 찾는 우선순위 모델로 다룬다. 예를 들어 이미 투자 중이어도 비상금이 없으면 비상금 단계가 우선이다.

## MVP 우선순위

1. 간편 입력 기반 자산 단계 진단
2. 주간 챌린지 추천
3. 챌린지별 짧은 학습 콘텐츠
4. 단계별 실행 가이드와 외부 링크
5. 완료 체크, 효과 계산, AI 회고
6. 스트릭과 성장 기록
7. 기본 지표 수집: 온보딩 완료율, 미션 수락률, 미션 완료율, 리텐션, AI 답변 만족도

## MVP 선별 원칙

새 기능은 아래 조건 중 하나 이상에 직접 기여할 때만 MVP에 넣는다.

- 챌린지 완료율을 높인다.
- 다음 주 재방문율을 높인다.
- 사용자가 금융 용어와 절차를 덜 부담스럽게 느끼게 한다.
- 사용자가 실제 금융 행동을 했는지 더 명확히 기록하게 한다.
- 개인정보와 금융정보 입력 부담을 줄인다.
- 지난 방문 이후 변화나 다음 행동을 명확히 보여줘 재방문 이유를 만든다.

초기 MVP에서 제외할 것:

- 인스타툰, AI 캐릭터 숏폼 같은 제작 부담이 큰 콘텐츠 포맷
- 친구 초대, SNS 공유 자동화 같은 바이럴 기능
- 복잡한 상품 가입 플로우나 금융상품 추천
- 계좌 연동, 증빙 자동 확인, 정교한 포트폴리오 분석
- ETF 구성 종목/수수료 검색 같은 투자 탐색 기능. 단, 투자 기초 단계의 후순위 실험 후보로 기록한다.
- 자산 컨설턴트 중개, Advanced 자산 분석 구독 같은 고비용/고책임 BM은 MVP 이후로 둔다.

## 리텐션 원칙

- 매일 접속을 강요하기보다 주간 금융 행동을 완료하게 한다.
- 재방문 화면에는 지난 방문 이후 변화, 보류한 챌린지, 다음 행동 1개를 우선 보여준다.
- 푸시/CRM은 온보딩 미완료, 챌린지 미완료, 보류 후 재시도, 월급일, 주간 요약처럼 사용자가 필요성을 느끼는 시점에만 사용한다.
- 무작위 보상이나 불안 자극은 MVP 기본 전략으로 쓰지 않는다.

## 챌린지 라이브러리 원칙

- 전체 후보 풀은 `shared/challenges/challenge-library-v1.yaml`에서 관리한다.
- 실제 구현할 챌린지는 `shared/challenges/mvp-challenges.yaml` 수준으로 상세화한 뒤 앱에 넣는다.
- 학습만 있는 항목은 독립 챌린지로 만들지 말고 `whyContent`, 실행 스텝, SRS 카드로 흡수한다.
- 실매수, 특정 종목 판단, 유망 산업 선택 유도 항목은 격리 목록으로 유지하고 추천 엔진 후보에 넣지 않는다.
- 정책, 세금, 청약 챌린지는 `sourceList`, `validUntil`, `sourceUpdatedAt`을 채운 뒤 배포한다.

## 기술 방향

- 모바일 앱: Flutter, Dart, Riverpod, Go Router
- 백엔드: Spring Boot, Java 17, PostgreSQL/Supabase
- AI 백엔드: FastAPI, Python, LangChain 또는 LlamaIndex
- AI 기능: 입력 구조화, 진단 설명, 챌린지 개인화, 금융 개념 설명, 회고 대화
- RAG 자료: 금융감독원, 국세청, 서민금융진흥원 등 공신력 있는 기관 자료
- 운영/분석: Firebase, Sentry, Mixpanel

## 저장소 작업 기준

- 이 저장소의 루트는 `didim/`이다.
- 제품 문서는 `docs/`에 둔다.
- Flutter 앱은 `frontend/`에 둔다.
- Spring Boot API는 `backend/`에 둔다.
- FastAPI/RAG 서비스는 `ai/`에 둔다.
- 앱, 백엔드, AI가 함께 쓰는 계약과 샘플 데이터는 `shared/`에 둔다.
- 특정 영역 작업을 요청받으면 해당 폴더 밖의 코드는 수정하지 않는다. 필요한 연결점은 `shared/`나 문서에 TODO로 남긴다.
- 글로벌 Codex 설정인 `~/.codex/config.toml`은 프로젝트 파일이 아니므로 사용자가 명시적으로 요청할 때만 다룬다.

## 안전 원칙

- 특정 금융상품 가입을 대신하거나 투자 판단을 단정하지 않는다.
- 정보 제공, 실행 가이드, 학습 코칭의 범위를 명확히 한다.
- 정책, 세금, 금융상품 정보는 날짜와 출처를 함께 다룬다.
- 개인 민감정보 수집은 최소화하고, 초기 MVP는 계좌 연동 없이 자가 입력으로 시작한다.
- 민감한 원본 금융정보보다 상태값과 범주형 데이터를 우선 저장한다.
- 추천 결과에는 룰 근거, 사용된 입력, 신뢰도 또는 검증 상태를 남긴다.
- 금융상품 제휴와 CPA는 사용자 신뢰를 해치지 않는 방식으로 후순위 검증한다.
