# 09. Open Questions

## 제품

- 첫 챌린지는 모든 사용자에게 동일하게 줄 것인가, 진단 결과에 따라 바로 달리 줄 것인가?
- ~~사용자가 챌린지를 완료했다는 증거를 어떻게 확인할 것인가?~~ → 자가검증 게이트 + 선택 증빙 보너스로 결정 ([08-decisions.md](./08-decisions.md) 2026-07-12)
- 자가검증 게이트가 실행 완료율을 얼마나 낮추는가? 이탈 폭이 크면 어느 단계를 줄일 것인가?
- 인증샷 첨부율이 낮으면 보너스 설계를 어떻게 조정할 것인가?
- 실측 입력의 이상치(허위 과대 입력)를 어떻게 다룰 것인가? 정합성 가드(1억 미만)로 충분한가?
- 금융 행동이 실패했거나 보류된 경우에도 학습 성과로 인정할 것인가?
- 챌린지 난이도와 주기를 사용자가 직접 조절할 수 있게 할 것인가?
- 효과 계산 결과가 과장된 기대를 만들지 않게 어떤 표현을 사용할 것인가?
- 지도 오브젝트 연출을 9월 일정 안에서 어느 수준(정적 일러스트 vs 애니메이션)으로 구현할 것인가?
- 지역 진행 가중치와 해금 기준(40%)이 실제 사용 패턴에서 적절한가?
- 계획 완료와 실행 완료를 사용자에게 어떤 문구로 구분할 것인가?
- 보류 사유를 몇 개까지 제공해야 부담 없이 다시 시도하게 만들 수 있는가?
- 지난 방문 이후 변화 카드는 홈 첫 화면에서 어느 정도 크기로 보여줄 것인가?
- 주간 요약은 매주 고정 요일에 보여줄 것인가, 사용자 행동이 쌓였을 때 보여줄 것인가?

## 금융/규제

- 정보 제공과 금융상품 추천의 경계는 어디까지인가?
- CPA 제휴가 들어갈 때 사용자에게 어떤 방식으로 고지할 것인가?
- 연말정산, 정책금융, 청약 정보의 기준일과 변경사항을 어떻게 관리할 것인가?
- 면책 문구를 어느 화면에 어떤 톤으로 배치할 것인가?

## AI/RAG

- 공식 자료를 어떤 단위로 청크화할 것인가?
- 출처 품질 점수와 최신성을 어떻게 계산할 것인가?
- AI 응답이 출처와 맞지 않을 때 어떻게 차단할 것인가?
- 챌린지 개인화에서 LLM의 자유도를 어디까지 허용할 것인가?
- 추천 근거 로그를 사용자가 이해할 수 있는 표현과 내부 평가용 표현으로 어떻게 나눌 것인가?
- Prompt Caching이 실제 비용 절감에 도움이 되는 호출 패턴은 무엇인가?

## 개발

- Spring Boot와 FastAPI 사이의 책임 경계를 어떻게 나눌 것인가?
- Supabase를 인증까지 사용할 것인가, DB로만 사용할 것인가?
- Mixpanel/Firebase 이벤트 스키마는 어떤 이름 규칙을 사용할 것인가?
- CRM 푸시는 웹 우선 개발 기간 동안 어떤 채널(웹 푸시, 이메일, 인앱)로 검증할 것인가?

## 마케팅

- 첫 유입 채널은 커뮤니티, 콘텐츠, 학교/부트캠프 제휴 중 무엇을 우선할 것인가?
- 랜딩페이지 핵심 메시지는 금융 불안 해소와 실행 챌린지 중 무엇을 전면에 둘 것인가?
- 베타 사용자는 몇 명까지 수동 코칭을 병행할 것인가?
- 카드뉴스, 진단형 콘텐츠, 챌린지 미리보기 중 어떤 포맷이 실제 챌린지 시작률이 가장 높은가?
- 청년 정책 체크 콘텐츠가 첫 챌린지 시작률을 높이는가?
- CRM 메시지는 푸시, 이메일, 인앱 중 어떤 채널부터 검증할 것인가?
- 자산 컨설턴트 중개 아이디어는 어떤 법적 검토와 신뢰 검증을 통과해야 실험할 수 있는가?
