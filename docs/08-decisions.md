## 2026-07-07: MVP는 완료율, 재방문율, 사용 편의성에 직접 기여하는 기능만 포함한다

- 기대 효과: 앱의 핵심 가설인 "금융 정보를 행동으로 바꿀 수 있는가"를 빠르게 검증한다.
- 후속 검토: 첫 챌린지 완료율과 2주차 재방문율을 기준으로 후순위 기능을 다시 평가한다.

## 2026-07-09: 챌린지 완료 보상은 금융 여정 지도, 배지, 완료 리액션으로 구성한다

- 배경: 기획심의에서 보상과 재방문 요소 부족이 지적되었다. 무작위 보상 금지 결정은 유지하면서 완료 성취감과 재방문 이유를 만드는 장치가 필요하다.
- 결정: 포인트·랜덤 보상 대신 ① 확보효과 이중 카운터(지킨 돈/예약된 돈) ② 금융 여정 지도(챌린지 완료 = 지도 오브젝트 변화, 출발지+3지역으로 시작) ③ 소수 배지(행동 5, 지역 3, 꾸준함 2, 성장 기록형) ④ 5~8초 완료 리액션 ⑤ 주간 스트릭(성실한 보류도 유지로 인정, 끊겨도 비초기화)으로 구성한다. 상세는 [16-gamification.md](./16-gamification.md).
- 제외: 포인트 상점, 랭킹·리더보드, 캐릭터 성장(후순위), 일일 출석 보상.
- 기대 효과: 진짜 돈과 지도 변화가 재방문 이유가 되어, 신뢰 포지션을 해치지 않고 리텐션을 만든다.
- 후속 검토: 완료 리액션의 다음 챌린지 수락률, 지도 조회 후 챌린지 시작률, 주간 스트릭 복구율, 2주차 재방문율을 측정해 지도 완성 자체가 보상으로 작동하는지 검증한다.

## 2026-07-13: 챌린지 외 상시 '도구' 레이어를 도입한다 (가계부 → 디지털 월세 → 카드 재테크)

- 배경: 챌린지는 주 1회 리듬이라 앱을 매일 열 이유가 약하다. 사용자가 상시 사용하는 도구 기능 3종(가계부, 디지털 월세(구독료) 관리, 카드 재테크 관리)을 요청했다.
- 결정: 웹 메뉴바에서 진입하는 '도구' 레이어를 순차 도입한다. 1차로 가계부를 구현했다 — 자가 입력(계좌 연동 없음), shared_preferences(웹 localStorage) 로컬 저장, 월 요약·카테고리별 지출·내역 관리. 디딤의 문제의식("가계부 앱은 기록 피로도가 높다", docs/00)과 충돌하지 않도록 기록 강제형이 아니라 챌린지 실측 입력을 뒷받침하는 가벼운 보조 도구로 포지셔닝한다.
- 제외: 계좌 연동 자동 기록, 지출 자동 분류, 예산 초과 푸시. 디지털 월세·카드 재테크는 다음 차수.
- 기대 효과: 재방문 이유가 주 1회(챌린지)에서 상시(도구)로 확장되고, 가계부 데이터가 챌린지 추천(고정비·무지출)의 재료가 된다.
- 후속 검토: 가계부 진입률과 기록 지속률, 가계부 사용자와 챌린지 완료율의 상관을 본다.

## 2026-07-12: 챌린지 실행 완료는 자가검증 게이트로 확인하고, 증빙은 선택 보너스로만 쓴다

- 배경: 실행 완료가 버튼 한 번으로 끝나 완료 검증이 사실상 없었다. 2026-07-07 결정대로 증빙 자동 확인은 MVP 제외를 유지하되, 09번 미해결 질문 "완료 증거를 어떻게 확인할 것인가"에 답이 필요했다.
- 결정: 실행 완료는 3단 자가검증 게이트(스텝 전체 체크 → 완료 확인 질문 전원 긍정 → 회고 1문 입력)를 통과해야 기록된다. 질문에 긍정하지 못하면 계획 완료(50%)로 전환을 제안한다. 인증샷 첨부는 선택사항이며, 보상은 배지 「기록으로 남긴 실행」(첫 1회)과 완료 리액션 보너스 표시로 한정한다. 사진은 기기에서만 잠깐 쓰이고 저장·전송하지 않으며 첨부 여부만 기록한다. 상세는 [02-challenge-system.md](./02-challenge-system.md) 완료 검증 방식.
- 제외: 증빙 필수화, 증빙 자동 검증(OCR·계좌 연동), 증빙 첨부의 금액(지킨 돈/예약된 돈) 가산, 증빙 이미지 영구 저장.
- 기대 효과: 개인정보 수집 없이 실행 완료의 신뢰도를 높이고, 회고 데이터가 쌓여 다음 추천의 근거가 된다. 금액 트랙의 진정성 원칙은 유지된다.
- 후속 검토: 게이트 도입 후 실행 완료율 변화(이탈 폭), 인증샷 첨부율, 회고 입력 품질을 측정한다.

## 2026-07-12: 확보효과는 고정 예시가 아니라 완료 시 실측 입력으로 계산한다

- 배경: 자가검증 게이트도 거짓 답변이 가능하다는 문제 제기. 계좌 연동 같은 완전한 검증은 규제·비용으로 로드맵에 있고, 지금은 거짓의 비용을 높이고 이득을 없애는 설계가 필요했다.
- 결정: 금액형 챌린지는 완료 확인 시트에서 실제 수치(해지 구독료, 자동이체 금액, 월세 등)를 입력해야 실행 완료를 확정할 수 있고, 확보효과는 `impactCalculation` 산식대로 입력값 × 계수로 그 자리에서 계산한다. 고정 예시 금액(118,800원 등)은 제거한다. 입력값은 1원 이상 1억 미만 정합성 가드를 거친다. 거짓 수치를 지어내면 본인 자산 기록이 오염되므로 거짓 유인이 급감한다. 검증 강도 사다리(후속 확인 → AI 인증샷 판독 → 계좌 연동)는 [02-challenge-system.md](./02-challenge-system.md)에 정리했고, 현금성 리워드·리더보드 도입 시 상위 단계 검증이 선행 조건이다.
- 제외: 이상치 자동 판별(통계 기반), 입력값 서버 검증, 후속 확인·AI 판독·계좌 연동의 즉시 구현.
- 기대 효과: 완료 데이터의 신뢰도와 확보효과 카운터의 진정성이 함께 올라가고, 산식 입력 데이터가 쌓여 개인화 추천의 재료가 된다.
- 후속 검토: 실측 입력 단계에서의 이탈률, 입력값 분포와 이상치(허위 과대 입력) 비율을 측정한다.

## 2026-07-12: 보상 보강은 즉시 가시화(지도 장면)와 리듬 유지(주간 스트릭)에 정렬한다

- 배경: 보상 시스템 확장 검토에서 벤치마크(Duolingo·Fortune City·Ant Forest·Strava·Forest vs Long Game·Yotta·Cleo)를 분석했다. 성공 사례의 공통 패턴은 ① 최소 행동의 즉시 가시화 ② 손실회피+안전망 ③ 누적물의 정체성화 ④ 실물·사회적 전환의 신뢰 ⑤ 가벼운 상호 인정이며, 실패 사례의 공통 원인은 외생 보상 경제 붕괴와 재미 프레임의 리스크 은폐(규제·신뢰 붕괴, 예: Cleo FTC $17M 합의)다. 기존 결정(무작위 보상·포인트 상점·랭킹 금지)과 정합했고, 격차는 설계만 있고 구현이 없던 지도 시각화와 주간 스트릭이었다.
- 결정: ① 금융 여정 지도를 마일스톤(20/40/60/80/100%) 이모지 장면으로 시각화하고 완료 리액션에서 새 오브젝트 등장을 연출한다 (9월 일러스트/애니메이션 결정 전 플레이스홀더). ② 주간 스트릭 카드를 홈에 추가한다 — 주 1개 이상 완료 시 유지, 성실한 보류도 유지 인정, 끊겨도 초기화하지 않음, 2주 루틴 배지 연동.
- 제외(재확인): 실물·현금 리워드(외생 보상 붕괴 + 규제 리스크 + 진정성 원칙), 또래 비교·랭킹·리더보드(비교 대상은 과거의 나, 도입하려면 강검증 선행 — 2026-07-12 검증 결정 참조).
- 다음 후보: shareSafe 규칙 기반 완료 카드 공유(가벼운 상호 인정). 베타 검토: Ant Forest식 이타적 실물 전환(가상 성과 → 검증 가능한 실물 기여, 신뢰할 수 있는 파트너 필요).
- 후속 검토: 스트릭 카드 유무별 2주차 재방문율, 지도 조회 후 챌린지 시작률 (docs/06 지표 연계).

## 2026-07-09: 서비스 웹은 Flutter 웹 타깃으로, 마케팅 랜딩은 별도 영역으로 만든다

- 배경: 웹과 앱을 한 프로젝트에서 만들지 결정이 필요했다. Flutter는 단일 코드베이스로 모바일과 웹을 함께 빌드할 수 있지만, Flutter Web은 캔버스 렌더링이라 SEO와 초기 로딩에 약하다.
- 결정: 서비스 앱은 `frontend/` Flutter 단일 코드베이스에서 android, ios, web을 함께 빌드한다(프로젝트명 `didim`, Riverpod + Go Router). SEO가 필요한 마케팅 랜딩페이지(간이 진단 + 베타 신청)는 Flutter Web으로 만들지 않고, 필요해지는 시점에 별도 `web/` 영역(Next.js 또는 정적 페이지)으로 만든다.
- 기대 효과: 코드 100% 공유로 유지비를 최소화하고, 웹 타깃을 베타 배포·데모 채널로 활용한다. 랜딩의 검색·공유 유입 목적은 훼손하지 않는다.
- 후속 검토: Flutter 앱 구조(feature-first vs layer-first)는 첫 기능 구현 전에 결정한다. 랜딩 제작 시점은 베타 모집 일정에 맞춘다.

## 2026-07-09: 웹 타깃을 먼저 개발하고 모바일 배포는 뒤로 미룬다

- 배경: 웹과 앱 중 웹을 먼저 만들기로 했다. 웹은 앱스토어 심사 없이 즉시 배포·수정할 수 있어 베타 검증 속도가 빠르다.
- 결정: 개발과 검증은 Flutter 웹 타깃(`flutter run -d chrome`, `flutter build web`) 기준으로 진행한다. 코드베이스는 동일하므로 모바일 배포는 베타 이후 빌드만 추가한다. 앱 구조는 feature-first(`lib/features/<기능>/`)로 하고, 데이터 계약은 `shared/` 파일을 기준으로 `lib/data/`에 둔다.
- 기대 효과: 배포 마찰 없이 첫 챌린지 완료율과 재방문율을 빠르게 검증하고, 모바일 전환 비용은 0에 가깝게 유지한다.
- 후속 검토: 베타 사용자 피드백에서 모바일 필요 신호(홈 화면 추가, 푸시 요구)가 크면 모바일 빌드 시점을 앞당긴다. 웹 푸시 제약은 CRM 채널 결정 시 함께 검토한다.

## 2026-07-07: 챌린지 라이브러리 v1은 후보 풀과 상세 구현 파일을 분리한다

- 배경: 새 챌린지 리스트는 온보딩 2개, MVP 코어 16개, 확장, 흡수, 격리·제외 항목을 포함한다. 기존 `mvp-challenges.yaml`에는 이미 상세화된 초기 구현 챌린지 6개가 있다.
