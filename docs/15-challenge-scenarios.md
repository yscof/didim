# 15. Challenge Scenarios (MVP 코어 16 상세)

## 목적

[14-challenge-library.md](./14-challenge-library.md)의 MVP 코어 16개를 실행 시나리오 수준으로 상세화한 기획 문서다. 구조화 데이터는 [shared/challenges/mvp-core-challenges.yaml](../shared/challenges/mvp-core-challenges.yaml)에 있고, 이 문서는 템플릿·설계 규칙·챌린지별 핵심 설계 결정을 설명한다.

상세화의 템플릿은 원본 스펙 문서 `didim_challenge_spec_tax.md`(M-13 "13월의 월급 예약하기", `tax_prebook_v1`)다. 16개 전부 같은 골격을 따른다.

## S2 실행 시나리오 템플릿

모든 챌린지는 다섯 스텝을 따른다.

1. **훅 (30초)** — 코치가 상식을 뒤집는 한 문장으로 시작한다. 예: "연말정산은 연말에 하는 게 아니에요." 훅은 `coachHook` 필드로 관리한다.
2. **입력/스캔** — 문항 응답 또는 간단 입력만으로 진행한다. 조회·서류·민감 증빙을 요구하지 않고, 모든 문항은 "대략"으로 답해도 충분함을 명시한다. 온보딩 응답(`coachContextUsed`)이 있으면 재사용해 문항을 줄인다.
3. **결과 (아하 모먼트)** — 개인화된 숫자 하나를 보여준다. "내 진짜 가처분액", "내년 현금 증가 예상액"처럼 발견 자체가 보상이 되게 한다.
4. **첫 행동 1개** — 결과에서 파생되는 행동 후보 중 **이번 주에 딱 하나만** 고르게 한다. 전부 하려다 미션이 비대해지는 것을 막고, 나머지는 후속 미션으로 자동 예약한다(모(母) 챌린지 패턴, 아래 참조).
5. **완료 확인** — 유형별 게이트(아래)와 회고 1문, 보류 시 사유 기록과 재시도 예약.

## 설계 규칙

### 확보효과(gain) 라벨 4종

기획심의 피드백대로 고정 금액이 아닌 산식으로 계산하되, 신뢰성을 위해 라벨을 구분한다.

| 라벨 | 의미 | 실적 카운터 |
|---|---|---|
| `discovery` | 발견 (금액 아님, 아하 모먼트) | 미합산 |
| `estimated` | 응답 기반 추정액. "최대" 표기 금지, 연도 표기 필수 | 별도 트랙 ("예약된 돈") |
| `annualized` | 월 절감·저축액 x 12. "연 환산" 라벨 명시 | 별도 트랙 |
| `realized` | 실제 아낀·모은 금액 (무지출, 배달 감축, 보류 후 미구매) | **실적 합산** |

실발생만 실적에 합산하는 것은 M-13 스펙 §6의 보상 원칙("실발생과 구분, 안심증서 진정성 원칙")을 전 챌린지로 일반화한 것이다.

### 유형별 완료 게이트

| 유형 | 게이트 | 해당 챌린지 |
|---|---|---|
| A 즉시완료 | 앱 안 결과 확인 + 회고 1문 | M-1, 2, 4, 9, 12, 16 |
| B 실행 | 첫 행동 자가체크 + 회고 게이트 | M-3, 5, 10, 11, 13, 14, 15 |
| C 습관 | 기간 내 체크인 누적 + 종료 회고 | M-6, 7, 8 |

### 모(母) 챌린지 패턴

스캔형 챌린지(M-12 놓치는 돈 스캔, M-13 연말정산)는 결과의 미실행 항목이 다음 주부터 개별 후속 미션으로 자동 예약된다. 챌린지 하나가 3~4주치 미션과 시즌 재방문 트리거를 심는 구조로, 리텐션 설계와 직결된다. 후속 미션은 YAML의 `followupMissions` 필드로 관리한다.

### 히어로 항목

스캔형 챌린지는 "감면율 크고, 실행 쉽고, 인지도 낮은" 항목 하나를 히어로로 정해 결과 최상단에 노출한다. M-13의 히어로는 **중소기업 취업자 소득세 감면**(소득세 90% 감면, 신청서 1장)이다. `hero_item_shown → acted` 전환율로 가설을 검증한다.

### 빈손 방지

어떤 분기로 빠져도 사용자가 빈손으로 나가지 않게 한다.

- 해당 항목 0개 (M-12) → 상황 변경 시 재스캔 트리거 등록
- 프리랜서 (M-13) → 5월 종합소득세 버전 예약 후 종료
- 해지할 구독 없음 (M-5) → 총 구독액 확인만으로 완료 + 칭찬
- 청약 소득공제 비대상 (M-15) → "가입 기간 자체가 자산" 안내

## M-13 상세 스펙 요약 (템플릿 원본)

원본 `didim_challenge_spec_tax.md`의 핵심 결정만 요약한다. 전체 코치 대사 세트·시즌 이원화 표는 원본 참조.

- **"정산"이 아니라 "예약"** — 승부는 연중이라는 프레임. 챌린지명부터 훅까지 일관.
- **스캔 + 첫 행동 1개** — 혜택 5종을 다 실행하면 미션 비대화. 스캔으로 지도를 만들고 이번 주 1개만.
- **히어로 = 중기 취업자 소득세 감면** — 감면율 최대 x 실행 최소 x 인지 최저.
- **시즌 이원화** — 연중형(3~10월) "예약" / 시즌형(11~2월) "채점+막차". 10월 말·1월 시즌 트리거가 연 3회 재방문을 만든다.
- **엣지케이스** — 프리랜서(5월 종소세 예약), 미취업(취업 시 재트리거), 이미 신청(완료+칭찬), 회사 규모 모름(확인 질문 문구까지 제공).
- **측정** — 핵심 학습 질문 두 개: ① 히어로 항목 인지→행동 전환율 ② 모 챌린지의 후속 미션 소화율.

세율·한도는 2026.01 기준 지식으로 작성됐으므로 출시 전 국세청 최신 고시로 재확인해야 한다. `validUntil`·`sourceUpdatedAt` 필드가 그 장치다.

## 기존 상세 6개와의 관계

[mvp-challenges.yaml](../shared/challenges/mvp-challenges.yaml)의 기존 6개는 초기 구현 슬라이스로 유지한다. 겹치는 항목의 관계:

- `housing-subscription-check` (상태 점검) → `mvp-housing-subscription-open`의 "이미 보유" 엣지케이스로 흡수
- `fixed-expense-cut-one` → `mvp-subscription-audit-cancel-one`이 구독 특화로 구체화한 버전
- `emergency-fund-goal` / `payday-auto-transfer-plan` → `mvp-emergency-fund-goal` / `mvp-emergency-fund-account-auto-transfer`가 고용 형태 개인화·통장 분리를 더한 버전

첫 배포 챌린지 세트를 확정할 때 두 파일 중 어느 스키마로 통합할지 결정한다 ([09-open-questions.md](./09-open-questions.md) 후보).

## 남은 작업

1. O-1 소비 캐릭터 테스트 문항 8개 + 캐릭터 6종 확정
2. gain 산식 기준값 출처 확정 (평균 배달비, 평균 일지출 등 — 안심증서·보상 스택 공통 사용)
3. M-12·M-13·M-15의 세율·한도 최신 고시 재검증 (`sourceUpdatedAt` 갱신)
4. M-13 시즌 트리거(10월 말·1월)를 [shared/crm-triggers.yaml](../shared/crm-triggers.yaml)에 연결
5. 측정 이벤트(`events`)를 [shared/events.md](../shared/events.md) 스키마와 정합화
